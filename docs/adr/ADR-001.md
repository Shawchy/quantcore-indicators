# ADR-001: 数据中台架构选择

## 状态
✅ **已采纳** (2026-03-28)

## 背景

### 问题陈述
在量化交易系统开发过程中，面临以下挑战：

1. **数据源多样化**: 需要接入多个数据源 (EFinance、AkShare、Baostock、TickFlow)，每个数据源 API 接口、数据格式、质量不同
2. **前端耦合**: 前端代码需要感知数据源差异，导致代码复杂、维护困难
3. **数据质量不一**: 不同数据源返回的数据格式、字段命名、数据质量存在差异
4. **故障处理**: 单个数据源故障时，缺乏自动降级机制
5. **扩展困难**: 新增数据源需要修改大量业务代码

### 需求分析
- 前端无需感知数据源差异
- 统一数据格式和质量标准
- 自动选择最优数据源
- 故障时自动降级切换
- 易于扩展新数据源

## 决策

### 选择方案
**采用"轻量化数据中台"架构**，作为数据源和业务层之间的中间层。

### 架构层次

```
┌─────────────────────────────────────────────────┐
│  前端层 (Frontend)                               │
│  - React + TypeScript                           │
│  - 统一 API 调用                                 │
└──────────────────┬──────────────────────────────┘
                   │
┌──────────────────▼──────────────────────────────┐
│  API 网关层 (FastAPI Endpoints)                  │
│  - 统一 REST API                                │
│  - 参数验证                                     │
│  - JWT 认证                                     │
└──────────────────┬──────────────────────────────┘
                   │
┌──────────────────▼──────────────────────────────┐
│  数据中台调度层 (Data Middleware) ⭐核心         │
├─────────────────────────────────────────────────┤
│  1. DataSourceFactory - 智能路由                │
│     - 主备源选择                                │
│     - 故障降级                                  │
├─────────────────────────────────────────────────┤
│  2. UnifiedDataAdapter - 统一适配器             │
│     - 数据标准化清洗                            │
│     - 质量验证                                  │
│     - 指标计算                                  │
├─────────────────────────────────────────────────┤
│  3. StorageRouter - 存储路由                    │
│     - SQLite (热数据)                           │
│     - Parquet (温数据)                          │
│     - Cache (LRU 缓存)                          │
└──────────────────┬──────────────────────────────┘
                   │
┌──────────────────▼──────────────────────────────┐
│  数据源适配层 (Data Source Adapters)             │
│  - EFinanceAdapter (主力)                        │
│  - AkShareAdapter (主力)                         │
│  - BaostockAdapter (主力)                        │
│  - TickFlowAdapter (可选)                        │
└─────────────────────────────────────────────────┘
```

### 核心组件

#### 1. DataSourceFactory
- **位置**: `backend/app/adapters/factory.py`
- **职责**: 数据源工厂，智能路由选择
- **特性**:
  - 支持优先级配置
  - 自动故障转移
  - 统一接口调用

#### 2. UnifiedDataAdapter
- **位置**: `backend/app/adapters/unified_adapter.py`
- **职责**: 统一数据适配器基类
- **特性**:
  - 统一数据模型 (UnifiedKLine)
  - 数据标准化清洗
  - 质量验证
  - 技术指标计算
  - 跨数据源校验

#### 3. StorageRouter
- **位置**: `backend/app/storage/storage_router.py`
- **职责**: 智能存储路由
- **策略**:
  - <90 天：SQLite (热数据)
  - >90 天：Parquet (温数据)
  - 实时：LRU Cache

## 后果

### 正面影响

1. **彻底解耦**
   - ✅ 前端无需感知数据源差异
   - ✅ 业务层无需关心数据获取细节
   - ✅ 数据源可独立替换和扩展

2. **标准化**
   - ✅ 统一数据模型 (100+ Pydantic 模型)
   - ✅ 统一清洗流程 (DataNormalizer)
   - ✅ 统一存储接口 (StorageRouter)

3. **智能化**
   - ✅ 自动选择最优数据源
   - ✅ 故障自动降级
   - ✅ 参数自动适配

4. **高可用**
   - ✅ 多数据源冗余
   - ✅ 自动故障转移
   - ✅ 健康检查监控

5. **易扩展**
   - ✅ 新增数据源只需添加适配器
   - ✅ 新增业务只需添加 API 端点
   - ✅ 支持多策略、多终端

### 负面影响

1. **开发成本**
   - ❌ 初期需要搭建完整架构
   - ❌ 需要编写多个适配器
   - ❌ 需要维护统一模型

2. **性能开销**
   - ❌ 增加了一层中间调用
   - ❌ 数据标准化需要额外处理时间
   - ⚠️ 影响较小 (<10ms)

3. **学习曲线**
   - ❌ 新开发者需要理解架构设计
   - ❌ 需要掌握多个组件的协作

### 权衡分析

**为什么选择自研而不是使用现成方案？**

| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| **自研中台** | 完全定制、可控性强 | 开发成本高 | ✅ 选择 |
| **Tushare** | 接口统一、文档完善 | 需要付费、依赖外部 | ❌ 放弃 |
| **单一数据源** | 简单直接 | 风险集中、易受限 | ❌ 放弃 |
